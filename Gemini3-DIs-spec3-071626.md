TFDA 醫療器材來源流向大數據與智慧稽查平台：全功能技術規格與架構設計說明書 (TFDA Medical Device Traceability Big Data & Intelligent Audit Platform: Technical Specification)
1. 執行摘要與設計哲學 (Executive Summary & Design Philosophy)
1.1 專案背景與痛點
根據中華民國《醫療器材管理法》第五十七條及《醫療器材來源流向資料建立及管理辦法》，第三等級植入性醫療器材（如乳房彌補物、人工心臟瓣膜、血管支架等）之輸入商、經銷商及醫療機構，必須建立並申報極其詳盡的來源與流向資料。然而，在後市場監管（Post-Market Surveillance）的實際執法中，主管機關與稽查人員常面臨以下嚴峻挑戰：
數據申報不對稱：進口商申報的銷售資料（Distribution Data）與醫院或診所申報的採購收貨資料（Purchase Data）採用不同的格式與編碼，且常因人工輸入漏失、時間滯後，導致「流向黑數」難以追查。
條碼（UDI）解析繁瑣：醫療器材單一識別系統（Unique Device Identification, UDI）包含產品識別碼（UDI-DI）與生產識別碼（UDI-PI，含批號、序號、有效期限等）。現行系統缺乏智慧拆解引擎，稽查人員需手動比對。
召回處置效率低下：當全球發布重大醫材不良反應召回通報（Recall）時，稽查人員無法在數秒內精確鎖定受影響的批號、序號已流向全台哪些醫學中心或醫美診所，容易錯失黃金封存時間。
執法資源不對等：全台醫材商與診所成千上萬，稽查人力有限，缺乏科學化、數據驅動的風險評級系統來引導「科技執法」。
1.2 系統願景
本系統旨在建構一座後市場安全防線，透過整合大數據雙向對帳、AI 語意理解（Gemini 3.1）、GIS 空間雷達、IoT 運輸追溯等前沿技術，打造全台首創的「TFDA 醫療器材來源流向大數據與智慧稽查平台」。
本平台能自動解算複雜的供應鏈關係，實時定位瑕疵產品，預警過期醫材的使用，並以智慧化的方式引導稽查決策。
1.3 視覺美學與品牌特徵：自然色調主題 (Natural Tones Design System)
本平台的 UI/UX 設計徹底摒棄傳統公家機關或高科技系統常見的「極客藍」或「刺眼暗黑漸變」，改用具備極高人文質感、護眼且沉穩的 「自然色調 (Natural Tones)」 品牌視覺體系：
主背景色 (Background)：使用溫潤的砂岩白 (stone-50 / #fafaf9) 作為大面積畫布，搭配極其柔和的溫帶暖灰 (stone-100 / #f5f5f4) 劃分區塊。
核心文字與首要元件 (Typography & Primary Elements)：選用沉穩、大氣的深石板黑 (stone-800 / #292524 與 stone-900 / #1c1917)，象徵執法的嚴謹與權威。
輔助強調色 (Accent Colors)：
警告與召回 (Alert/Recall)：採用略帶赤陶土色的暖赭紅 (amber-800 / #78350f 或暖赤色 rose-800 / #9f1239），取代刺眼的純紅。
合規與安全 (Compliance/Safe)：採用森林深綠 (emerald-800 / #065f46 與 emerald-50 的淡雅搭配），帶來生態、安全的和諧感。
字型配搭 (Typography Pairing)：
大標題與展示字體：選用兼具現代感與古典雅致的襯線字體（如 Playfair Display 或具備雕刻美感的 Outfit），營造宛若瑞士設計的現代美學。
正文與數據指標：採用高可讀性的無襯線字體 Inter，大數據表格與條碼序號則強制套用等寬字體 JetBrains Mono，確保每一個字元在稽查對帳時清晰無比。
微動效與材質 (Micro-interactions & Material)：元件邊框採用細膩極簡的 stone-200 實線，陰影採用極輕量、具空氣感的羽化投影（shadow-sm），按鈕配備平滑的縮放過渡（active:scale-95），展現工藝級的數位質感。

2.2 大數據流向生命週期
為保證大數據（數萬條 CSV 申報記錄）在瀏覽器中的瞬時解算，本平台建立了一套高效的數據生命週期流程：
資料導入（Ingestion）：
用戶上傳或黏貼銷售（Distribution）與採購（Purchase）CSV。
預設載入「曼陀矽膠填充義乳（衛署醫器輸字第019462號）」的大量真實流向歷史大數據。
正規化與解析（Normalization & Parsing）：
清洗逗號分隔符號（處理包裹在雙引號中的複雜中文品名與長型號文字）。
標準化日期格式為 YYYY-MM-DD 格式。
提取 UDI-DI（如 00081317026877）與 UDI-PI。
大腦級雙向勾稽（Reconciliation Hub）：
以「產品唯一序號（Serial Number）」或「產品批號（Lot Number）」為關聯金鑰（Join Key）。
將「申報業者（出貨端）」與「供應對象（收貨端）」進行雙向矩陣碰撞。
若某序號僅在銷售端出現而在採購端缺席，標記為 流失黑數。
若某序號僅在採購端出現而無經銷商銷售記錄，標記為 黑市/非法走私水貨。
若兩端皆有，但保存期限、有效期間、品名有出入，標記為 申報不符。
大數據速報與指標計算（Telemetry Summary）：
在記憶體（In-Memory）中快速運行 MapReduce 計算，生成：總申報筆數、涉及醫材商與醫療機構總數、雙向流向異常率。
3. 核心功能模組詳解 (Deep-Dive into Core Modules)
3.1 核心功能 1：快速回收（Recall）與緊急定向精準鎖定
當食藥署發布某一特定批號的「曼陀矽膠義乳」具有安全疑慮（例如：原廠通報製程不穩定，可能導致外殼破裂滲漏）時，稽查官必須立刻知道這些高風險義乳目前「落在哪一家醫院的哪一個倉庫，或是已經被植入哪位患者體內」。
3.1.1 檢索過濾器設計 (Search Matrix Filter)
檢索區域必須支持大數據快速索引。核心過濾欄位包括：
分銷商 ID (Distributor ID)：精確鎖定特定的經銷商（如 B00059：台灣愛力根、B00073：強生等）。
許可證號 (License No)：如 衛署醫器輸字第019462號。
型號 SKU (Model No)：如 SMPB340、SMHX350。
批號/序號 (Lot / Serial Number)：支持模糊與精確匹配。批號是生產批次（如 2098982），序號是單顆唯一碼（如 2098982097）。
交貨日期區間 (Date Zone Range)：限制數據檢索的時間窗口。
3.1.2 關聯比對演算法與動態呈現
系統以秒級速度檢索全部銷售記錄，將匹配到的每一顆義乳的流向狀態以高階視覺化表格展示。
召回狀態自動判定：
若產品批號完全等於受影響批次（預設瑕疵批號：2098982），其「當前流向狀態」欄位將顯示深紅色閃爍警示徽章：[ ⚠️ 嚴重心血管材料召回中 ]。
其餘無瑕疵的批次則顯示：[ 🟢 常態流通 ]。

設計理念：稽查人員面對複雜的對帳表格與法律條文，極需一個懂數據、懂法律的隨身助手。
技術實現：
集成最新的 @google/genai 開發包。
稽查官可以直接在對話方塊中使用白話文提問，例如：「請幫我分析資料庫中，有哪些醫療機構進貨量大於 5 顆，但雙向對帳完全不對稱？並告訴我如果去查扣，可以引用哪一條法規、開罰多少錢？」
提示詞工程（Prompt Engineering）：系統預先注入完整的《醫療器材管理法》罰則條款（包括第十九條、第五十七條、第七十條等），AI 回答時會精準引用法規，並將數據分析與法律責任相結合。

設計理念：矽膠義乳在極高溫下（如盛夏貨車無冷鏈保護、曝曬於 40°C 以上長達數小時），其外殼材料可能會發生化學降解或微量矽膠滲漏。
技術實現：
前端 Canvas 實時繪製 24 小時運輸溫度動態軌跡。

6.2 雙向勾稽解算演算法 (Bi-directional Join Algorithm)
在 React 前端或 Express 後端中，系統採用以下核心 TypeScript 演算法邏輯進行高效對帳：
code
TypeScript
interface AuditRecord {
  key: string; // 產品唯一序號或批號
  model: string;
  distRecord?: any;
  purchRecord?: any;
  conclusion: 'match' | 'mismatch_exp' | 'hospital_missing' | 'smuggled';
  penaltyCode: string;
}

function executeAudit(distList: any[], purchList: any[]): AuditRecord[] {
  const auditMap = new Map<string, AuditRecord>();

  // 1. 處理銷售申報庫
  distList.forEach(dist => {
    const key = dist.產品序號 || dist.產品批號;
    if (!key) return;
    
    auditMap.set(key, {
      key,
      model: dist.產品型號 || '',
      distRecord: dist,
      conclusion: 'hospital_missing', // 暫定為醫院漏報
      penaltyCode: '醫療器材管理法第57條：限期改善'
    });
  });

  // 2. 處理採購收貨庫
  purchList.forEach(purch => {
    const key = purch.產品序號 || purch.產品批號;
    if (!key) return;

    if (auditMap.has(key)) {
      const record = auditMap.get(key)!;
      record.purchRecord = purch;
      
      // 比對屬性：保存期限
      if (record.distRecord.保存期限 && purch.保存期限 && record.distRecord.保存期限 !== purch.保存期限) {
        record.conclusion = 'mismatch_exp';
        record.penaltyCode = '保存期限申報不一致：限期勘正';
      } else {
        record.conclusion = 'match';
        record.penaltyCode = '常態合規免處置';
      }
    } else {
      // 銷售端無此序號，但採購端有
      auditMap.set(key, {
        key,
        model: purch.產品型號 || '',
        purchRecord: purch,
        conclusion: 'smuggled', // 疑似黑市水貨
        penaltyCode: '查扣封存，依第70條移送地方衛生局刑事調查'
      });
    }
  });

  return Array.from(auditMap.values());
}

8. 潛在 Bug 防治、邊界條件與高性能優化 (Bug Prevention & Performance)
8.1 預防 React 常見無限重繪 (Infinite Re-render Prevention)
在處理解析 CSV、生成雙向勾稽結果、以及計算綜合風險分數等重度 CPU 運算時，必須極力避免因狀態更新觸發 React 元件 Body 重複執行大數據解算。
優化方案：
大數據集（distributionData、purchaseData）應存放在 useRef 中，或在狀態更新時使用 useMemo 對勾稽函數進行重度快取。
useEffect 的依賴陣列中，絕對不要傳入大數據 Object 本身，只傳入 Primitive Values（如 limit 數字、activeTab 字串、triggerTimestamp 戳記）。
8.2 殘缺數據的容錯（Handling Nulls & Corrupted CSV Rows）
在實際申報中，有些欄位極易缺漏。
序號空白與批號模糊匹配：部分醫材進貨時只有「批號」沒有「唯一序號」。在一致性勾稽中，若 產品序號 為空，系統會自動降級使用 產品批號 進行一對多勾稽，不會直接拋出錯誤或漏失該行。
日期格式不齊：有些業者填寫 2026/04/04，有些填寫 20260404。系統正則引擎會在正規化時統一轉化，若實在無法解析，將賦予 保存期限空白 的 Critical 評級，並在 AI 審計報告中高亮提醒。
8.3 大型上下文（Large Context）下的 Gemini Token 優化
如果一次性黏貼 50,000 條 CSV，Gemini 的 Token 消耗將會極快，甚至觸發 API 頻寬限制。
優化方案（Smart Token Slicing）：
在傳送給 Gemini 模型進行法規與大數據分析前，系統會先在前端/後端進行異常數據預先篩選（Anomaly Pre-filtering）。
我們只將「雙向不一致的紀錄（如 C07288 診所的流失黑數）」與「前 5 大高風險機構的統計概要」提取出來，組裝成緊湊的 JSON Snippet 傳送給 Gemini。
如此一來，既保證了 AI 能夠精確回答「哪裡有問題、該罰多少錢」，又將 Token 消耗降低了 98% 以上。
